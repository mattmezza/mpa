# Agents

A **agent** is a first-class, swappable identity the agent can take on — a fitness
coach, a finance assistant, a travel planner — without any bespoke code. It is the same
idea as the base `character`, made selectable.

An agent is exactly these things:

- its own **identity** — `character` (who the agent is + its tone & behaviour);
- an optional **agent name** it goes by (e.g. *Forge*), overriding the global assistant name;
- a **skill allowlist** — which skills it may load (empty = all);
- a **tool scope** — which function-tools are advertised (empty = all; `load_skill` is always available);
- a **voice** — a TTS voice override (empty = the configured default);
- a **secret scope** — which [vault](/docs/secrets) secrets it may use, enforced as a reference-only ACL (an agent may use a secret only if it is shared or named in this scope);
- **tool identities** — its own credentials and sessions for the external CLI tools (its own GitHub token, its own browser profile), so each agent acts as a distinct identity (see [Tool identities](#tool-identities));
- **email, calendar & contacts accounts** — which mail/calendar/contacts accounts it may use, at `read` or `read_write`, with an optional send identity (empty = no access; see [Email, calendar & contacts accounts](#email-calendar--contacts-accounts)).

When **no** other agent is active, the assistant uses the **default agent** — the configured
`character` (plus name and voice), edited at the top of the **Agents** tab — so first-run behaviour
is unchanged. The default agent is always present; custom agents are alternatives that replace it
when activated.

## Where agents live

Agents mirror the [skills](/docs/skills) pipeline:

- **Definitions** are stored in SQLite at `data/agents.db` (table `agents`), configurable
  via `agent.agents_db_path`.
- On first access they are **seeded** from any markdown files in `agents/` (configurable via
  `agent.agents_dir`). No agents ship by default — the directory is empty, so a fresh install
  starts with none; drop your own `.md` files there to auto-seed a gallery.
- The **active** agent is a single config value, `agent.active_agent`, in the config store
  (`data/config.db`). Empty = the default identity.

## Markdown format

An agent is a markdown file with YAML frontmatter. Identity lives in the frontmatter as block
scalars; any markdown body after the frontmatter is appended to `character` (which holds
both who the agent is and its tone).

```markdown
---
agent_name: Forge
role: Fitness coach
emoji: "🏋️"
voice: en-US-GuyNeural
skills: [scheduling, memory, weather]
tools: [send_message, create_calendar_event, manage_jobs, web_search]
secrets: []
tool_config:
  gh: {enabled: true}            # uses this agent's own GitHub token
  browser: {enabled: true, profile: forge}
character: |
  You are a strength & conditioning coach…

  ## Tone
  - Direct and motivating. Short, punchy messages.
---
```

> A legacy `personalia:` key is still accepted — on load it is folded into
> `character` (prepended), so older agent files keep working.

The file stem is the agent's **slug** (its identifier, e.g. `fitness-coach`). `role` is the
human label shown in the UI and injected into the prompt (*"acting as the Fitness coach
agent"*). `agent_name` is the name the assistant goes by while this agent is active.

## Managing agents (admin UI)

The **Agents** tab (next to *You*) shows the **default agent** at the top and lists every custom
agent as a card below, marking the active one. From there you can:

- **Use** an agent (activates it; applies live to new turns) or revert to **Default**;
- **edit** an agent in the editor — a **guided form** (identity, voice, skill + tool
  checkboxes, secret scope) with a **raw markdown** toggle for power users;
- **rename** an agent's slug in the editor — the new slug is cascaded to everything that
  references it (per-chat bindings, private memories, scheduled jobs, the active selection,
  and the `telegram:<slug>` bot channel), so no association is lost. An agent with its own
  bot needs an agent restart afterwards so the bot re-registers under the new slug;
- **create** or **delete** agents.

The page is responsive and touch-friendly: cards reflow to a single column at phone width.

### Seeding your own

No agents ship by default — you build them from scratch on the **Agents** tab (or in the
markdown format above). To pre-load a gallery, drop `.md` files into `agents/` (the
`agent.agents_dir`); each is seeded on first access.

Seeding is by file stem: a slug is seeded whenever no row by that name exists. Deleting or
renaming a seeded agent **tombstones its old slug**, so its file no longer re-seeds it — a
delete stays deleted and a rename moves the agent cleanly instead of leaving a duplicate copy
behind. Re-creating a slug (saving a new agent under it) clears the tombstone.

## Setup wizard

The setup wizard has a skippable **agent** step (after *identity*): keep the default general
assistant, pick one of any agents you've seeded, or skip and build agents later.

## Testing an agent from the terminal

The REPL can start with a specific agent active, so you can iterate on one without touching
the global setting:

```bash
make repl AGENT=<slug>
# or
uv run python -m core.repl --agent <slug>
```

An unknown name fails loudly and lists the available agents.

## How activation affects the prompt

When an agent is active, `_build_system_prompt` swaps in the agent's `character`,
narrows the advertised skills index to the allowlist (and refuses `load_skill`
for skills outside it), filters the function-tool set to the tool scope, and uses the
agent's voice for TTS. Switching applies to **new** turns; an open session keeps its
snapshotted identity until you send `/new`.

## Per-chat agents

By default the active agent is global. You can instead bind an agent to a single
conversation — it overrides the global one **only there**. Resolution runs a ladder in
`AgentCore._resolve_agent(channel, user_id, chat_id)`:

0. a **per-agent bot** — a `telegram:<agent>` channel binds straight to that agent
   (see [Bot-per-agent](#bot-per-agent)),
1. the per-chat binding for `(channel, user_id, chat_id)`,
2. the global active agent,
3. the default identity.

Bindings live in the `chat_agent` table and work on **any** channel (Telegram, WhatsApp,
REPL). They are set programmatically — by **bot-per-agent** routing and by Telegram
**topic auto-bind** (below) — rather than hand-picked per chat; the project is moving toward
**one agent = one agent** (a dedicated `telegram:<agent>` bot per identity), so the admin
UI no longer offers a per-chat agent dropdown. A binding applies on the next turn; the open
chat keeps its current identity until then (its messages are preserved).

### Telegram forum topics (optional)

Set `channels.telegram.topics_enabled: true` (or tick *Separate agent per forum topic* in
the Telegram channel setup) to treat each forum topic in a group as its own context. The
topic's `message_thread_id` is folded into the `chat_id`, so every topic gets isolated
history and its own agent binding; the group's General topic maps to the default context.
Name a topic after an agent (its name, agent name, or role) and it auto-binds when the topic
is created or renamed.

## Tool identities

By default every agent shares the system-wide tool configuration set in the **Tools**
tab — the same `GH_TOKEN`, the same browser sessions. So actions from *Hopper* (a coding
helper) and *Atlas* (a travel planner) all appear as the **owner's** GitHub user, and a
login one agent makes is visible to all of them.

**Tool identities** give each agent its own credentials and sessions for the external
CLI tools, so it acts as a distinct identity. In the agent editor, the **Tool identities**
card lists every system-enabled external tool with three states:

- **Inherit** — use the system-wide config (shared with the owner). This is the default and
  matches pre-existing behaviour, so nothing changes until you opt an agent in.
- **Enabled** — use *this agent's* own credentials/profile (configured below the toggle).
- **Disabled** — the tool is hidden from this agent's prompt and its credentials are
  stripped from the command environment (every `GH_TOKEN`/`GITHUB_TOKEN` variant), so the
  agent can never fall back to the owner's.

Supported tools:

| Tool | Per-agent identity |
|------|----------------------|
| `gh` (GitHub) | Pick the agent's identity explicitly with **GitHub identity → Personal Access Token or GitHub App** (raw frontmatter `gh: {enabled: true, auth: "pat"}` or `auth: "app"`). **PAT:** type a new token (stored encrypted in the **infra vault**, machine-key so it works headless and in [scheduled jobs](/docs/scheduler), namespaced `GH_TOKEN_<SLUG>`) or reuse an existing infra-vault secret (`gh: {auth: "pat", token_secret: SOME_INFRA_SECRET}`). **GitHub App:** give the agent its **own** app — `gh: {auth: "app", app_id: "123", installation_id: "456", private_key_secret: SOME_PEM_SECRET}` (paste the PEM as an infra secret first, then reference it) — so it runs as a distinct `<app>[bot]`. Leave the app fields blank to use the system-wide App bot. So you can run **multiple apps, one per agent**, mixed freely with PAT agents. An agent **never** borrows the owner's PAT. Optionally set an **allowed repos** list (`gh: {..., repos: [owner/a, owner/b]}`) — a best-effort `--repo` gate; the app installation's own repo selection is the hard boundary, and a subagent can only narrow this list, never widen it. |
| `browser` | The agent's own **profile** — an isolated Chromium session dir, so its logged-in sessions don't leak to other agents. Used by default (the agent need not pass `--profile`). Blank = the agent slug. |

How it flows into the agent: when an agent runs `run_command`, its tool environment is
swapped in for that call, and the `<tools>` section of the system prompt notes which
identity is active (*"`gh` is authenticated with this agent's own token"*). An agent
with **no** `tool_config` entry for a tool inherits the system config unchanged, so existing
setups keep working.

In raw markdown the field is `tool_config` (a map of `{tool: {enabled, …}}`); the GitHub
token itself is **never** stored in the agent doc — it lives only in the vault. Setting a
per-agent token requires a master key (configure one in the **Vaults** tab first).

> **Note:** per-agent tokens live in the infra vault, which has the same on-disk posture
> as the system `GH_TOKEN` (a machine key on the data volume), not the password-gated agent
> vault. That is a deliberate trade-off so the tokens are available headlessly; treat the
> data volume accordingly.
>
> Per-agent `gh` isolation assumes **token-based** auth. A shared `gh auth login` writes
> credentials to `~/.config/gh` (a shared HOME), which token env vars can't override — so
> authenticate `gh` via per-agent tokens here, not an interactive login.

## Email, calendar & contacts accounts

Each agent declares which email, calendar, and contacts **accounts** it may use, and at what
level ([#110](https://github.com/mattmezza/mpa/issues/110)). The accounts themselves live in the
**Email**, **Calendar**, and **Contacts** tabs (the registry); an agent simply **binds** to
some of them.

Two shapes of account, one model:

- **Agent-owned** — the agent has its own dedicated mailbox / calendar / address book (e.g. a
  cheap one-account-per-agent provider such as Purelymail). It sends, receives, schedules, and
  files contacts as itself.
- **User-owned (delegated)** — the agent is granted access to *your* personal or work account.
  Access is per-agent and per-account: agent A may have read-only on your personal inbox,
  agent B read/write on your work calendar, agent C nothing at all.

Each binding carries an **access level** and, for email, a **sender identity** flag:

- **read** — the agent may read that account only (list/search).
- **read/write** — it may also send email / create calendar events / add contacts through it.
- **sender identity** (email only, at most one) — the account the agent sends *from* by
  default; automatically read/write (you cannot send from a read-only account).

**An agent with no bindings has no email, calendar, or contacts access** — the safe default.
`send_email`, `reply_email`, `create_calendar_event`, `search_contacts`, and `create_contact`
route to the agent's bound account automatically: they default to the sender identity / a
writable calendar / a bound contacts book, so the model rarely passes an `account` at all. A
call to an unbound account, or a *write* (send / create) on a read-only binding, is refused with
a clear error. Which accounts the agent holds (names + levels, never credentials) is listed in
its per-turn prompt, so it routes without guessing and can answer *"what accounts does the
fitness agent have?"* on the go.

In the agent editor, the **Email, calendar & contacts accounts** card lists every configured
account with a **None / Read / Read & write** selector and — for email — a **sender** radio. In
raw markdown the frontmatter keys are:

```yaml
email_accounts:
  - account: fitness-agent      # a name from the Accounts → Email tab
    access_level: read_write
    is_sender_identity: true
  - account: personal           # delegated, read-only access to your inbox
    access_level: read
calendar_accounts:
  - account: fitness-agent
    access_level: read_write
contacts_accounts:
  - account: fitness-agent      # a name from the Accounts → Contacts tab (CardDAV/WebDAV)
    access_level: read_write
```

### The default agent

The default agent — active whenever **no** other agent is (the top of the **Agents** tab,
`agent.character`) — has its own bindings too, set there under *Email, calendar & contacts
accounts*. When you bind accounts there, the default agent routes and is enforced exactly like any
agent; leave them all on **None** and it stays **unscoped** (the legacy behaviour: it may use
any configured account, passing the account name explicitly). Changes apply live, no restart.

### Contacts (CardDAV / WebDAV)

Contacts speak CardDAV over plain WebDAV (`PROPFIND` to list, `GET` to read, `PUT` to create), so
they work against Purelymail's WebDAV address book, iCloud, Fastmail, and most work servers. Point
a CardDAV provider at your address-book URL (for Purelymail, your unique URL from
`purelymail.com/manage/cardDavUrl`). Google Contacts is supported for **reading** only (People API
scope). `create_contact` requires a `read_write` binding.

**Credentials never touch the model.** An email account's password lives in config (encrypted at
rest) or, better, in the [infra vault](/docs/secrets) referenced as `${vault:NAME}` — resolved
only when the Himalaya config is materialised, so it reaches the CLI but never the prompt, chat,
or logs. Calendar/contacts passwords are read from the (encrypted-at-rest) config store. Binding
levels are enforced at the tool boundary, and **subagents inherit-never-widen**: a subagent can
reach only accounts its parent identity holds, at an equal or lower level.

> **Upgrade note:** before #110 any agent could use any configured account. On the first start
> after upgrading, every existing agent that has no bindings yet is granted read/write access to
> all existing accounts (the first email account becomes its sender identity), so nothing breaks.
> Agents you create afterwards start with **no** access — bind accounts to them explicitly.

> **Test connection.** Adding an account in the **Accounts** tab (Email / Calendars / Contacts) offers provider
> presets (Purelymail, Gmail, Fastmail; generic CalDAV) and a **Test connection** button that
> verifies IMAP/SMTP/CalDAV/CardDAV reachability before you save — completable on a phone in under
> a minute.

## Bot-per-agent

Give an agent its own Telegram **bot token** and it becomes a separate contact in your
address book — its own name, emoji, and voice — addressable independently of every other
agent ([#29](https://github.com/mattmezza/mpa/issues/29)). Set it on the agent, not the
channel: the bot is part of the agent's identity.

- **Configure** — open the agent in the admin UI and fill in *Own Telegram bot* (token from
  [@BotFather](https://t.me/botfather), plus optional allowed user IDs — blank inherits the
  global Telegram allowlist). In raw markdown, the frontmatter keys are `bot_token` and
  `allowed_user_ids`. Leave the token blank and the agent stays reachable only through the
  default bot. **Restart the agent** after adding or changing a token.
- **How it routes** — each agent-bot registers as the channel `telegram:<agent>` and runs
  in the same process. That one composite name does three jobs: it **isolates history** (a
  Telegram DM has `chat_id == user_id` for every bot, so without it two bots' DMs with you
  would collide), it **routes approvals and proactive messages** back through the bot that
  fired them, and it **resolves the agent** with no lookup (rung 0 above). The default bot
  stays the bare `telegram` channel.
- **Scheduled jobs** — a job's `channel` accepts `telegram:<agent>` to deliver through that
  bot; bare `telegram` still targets the default bot, so existing jobs are unchanged.

Same process, N bots today; a separate deployment per agent (one token each) is a later
step that needs no further code.

## Roadmap

Agent resolution is deliberately funnelled through one seam,
`AgentCore._resolve_agent(channel, user_id, chat_id)`, so future work is additive:

- **Group multi-agent** ([#30](https://github.com/mattmezza/mpa/issues/30)) — several
  agent-bots in one group chat, taking turns. Builds directly on bot-per-agent.

The **secret scope** facet is now enforced by the [secrets vault](/docs/secrets) — an agent
uses a secret by reference (`{{secret:NAME}}`) only if it is shared or in scope, and values
never enter the model's context.
