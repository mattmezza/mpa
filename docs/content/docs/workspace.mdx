# Coding harness

By default the agent can *talk* about code but can't touch it — every change has to
be described in chat and applied by hand. The **coding harness** gives it a small set
of file tools so it can read, edit, search, and run commands on a real codebase
directly, like a built-in coding assistant.

It is deliberately minimal. Out of scope for this first iteration: AST/structural
editing, language-server integration, a plugin system, and a terminal/REPL.

## Tools

All paths are relative to the configured workspace root (or absolute *inside* it).

| Tool | Approval | What it does |
| --- | --- | --- |
| `read_file(path, offset=0, limit=100)` | pre-approved | Read a file (or a line range), returned with line numbers. Large files paginate via `offset`/`limit`. |
| `list_dir(path=".")` | pre-approved | List one level: `[{name, type, size}]`. |
| `grep(pattern, path=".", include="")` | pre-approved | Recursive regex search; `include` is a filename glob (e.g. `*.py`). Returns `[{file, line, content}]`. |
| `write_file(path, content)` | **asks first** | Write a file, creating parent directories. Overwrites. |
| `edit_file(path, old_string, new_string, multiple=false)` | **asks first** | Find-and-replace. `old_string` must match exactly and, unless `multiple` is true, be unique. |
| `run_command_in_dir(command, workdir=".")` | **asks first** | Run a shell command (linter, tests, build) in a workspace directory. Returns `stdout`, `stderr`, `exit_code`. |

Reads are pre-approved (`ALWAYS`); `write_file`/`edit_file` are write-actions that ask
for your approval on every call, surfaced through the same [permission](/docs/permissions)
flow as email/calendar/message sends. `run_command_in_dir` shares the `run_command`
permission rules: the hard `NEVER` rails always apply, a command you've globally marked
`ALWAYS` (e.g. `git log`) runs without a prompt, and anything else asks.

## Confinement

The **file** tools (`read_file`/`write_file`/`edit_file`/`list_dir`/`grep`) are confined
to the single **workspace directory** you configure. The check is `realpath`-based: a
path is resolved (following symlinks and collapsing `..`) and must land on the workspace
root or a descendant of it. So neither `../../etc/passwd` nor a symlink pointing outside
the workspace can escape — both are blocked (`NEVER`).

`run_command_in_dir` confines only the **working directory** it runs in — *not* the
command itself. The command runs through the shell with the full privileges of the humux
process, so it can still read paths outside the workspace (e.g. `cat /etc/passwd`). Its
real boundary is the permission gate (the `run_command` rules above), not the workspace.
Enable it only if you're comfortable with that.

There is **no default workspace**. With the harness disabled, or with a blank
directory, the tools are hidden from the agent entirely and the handlers refuse — the
agent cannot reach the filesystem until you point it at a directory.

## Configuration

Enable it under **Settings → Workspace** in the admin UI, or in `config.yml`:

```yaml
workspace:
  enabled: true
  directory: "~/projects/my-repo"   # the only directory the agent may touch
```

Config is applied live — the tool set is recomputed each turn, so a change takes effect
on the agent's next turn (no restart or `/new` needed). Each call is written to the agent
tool log on the **Logs** tab.

Like every tool, the harness can be scoped per [agent](/docs/agents) — grant it to
a "developer" agent and leave it off everywhere else.
