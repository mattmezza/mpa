# Web artifacts

When an answer is richer than a chat bubble can show — a monthly expense dashboard,
a chart comparing two options, an interactive checklist, a visual summary of GitHub
activity, a generated PDF report — the agent can publish it as a **web artifact**:
files it writes into the workspace, served at `/artifacts/<slug>/` on the admin app
and handed back to you as a link.

It is the agent's "give me a mini-site / document for X" capability. No nginx, no build
step — just a URL you can open on any device.

## How it works

There is no dedicated artifact tool. An artifact is plain files the agent writes under
`{workspace}/artifacts/<slug>/` using the [coding-harness](/docs/configuration#workspace)
`write_file` tool, and the admin app serves that directory at `/artifacts/<slug>/`:

- **A single page** — write a complete HTML document to `artifacts/<slug>/index.html`.
- **A multi-file site** — write `artifacts/<slug>/index.html` plus sibling `style.css`,
  `app.js`, images, etc. Files link each other with relative URLs (`href="style.css"`).
- **A generated file** — produce a PDF/image/slides on disk inside `artifacts/<slug>/`
  (e.g. with `pandoc` via `run_command_in_dir`) and link to it.

The page is then live at:

```
http://your-host:8000/artifacts/q3-report/
```

`<slug>` is agent-chosen — letters, digits, `-` and `_`. Set
[`HUMUX_BASE_URL`](/docs/configuration) to your reachable admin URL so the links work from
your phone, not just `localhost`.

Because artifacts are just workspace files, the **same** `write_file` / `edit_file` /
`list_dir` patterns the agent uses for code also publish and update artifacts — and the
agent can `read_file` one back before sharing. The capability is taught by the
`artifact-hosting` skill rather than a built-in tool.

### Requires the workspace harness

Artifacts live in the workspace, so publishing needs the
[workspace harness](/docs/configuration#workspace) enabled (it provides the `write_file`
tool and the directory). If the workspace is off, there is no place to write artifacts and
the `/artifacts/` route serves nothing.

### The complexity ladder

For HTML, the agent climbs only as high as the request needs:

| Need | What the agent reaches for |
|---|---|
| Quick report | Plain semantic HTML, no styling |
| Clean readable doc | A classless CSS framework (MVP.css / Water.css) via CDN |
| Branded / designed page | Custom CSS in `<style>`, or TailwindCSS v4 via its browser CDN build (`@tailwindcss/browser@4`) |
| Interactivity | Inline JS, or Alpine.js via CDN |

### Lifetime

There is **no TTL or automatic expiry** — artifacts are files, so they live until removed.
The agent overwrites a slug to update it in place, browses with `list_dir`, and cleans up
old ones as a filesystem operation. Manage them like any other workspace files.

## Security & hygiene

- **Public, guessable links.** The slug is agent-chosen, not an unguessable id, so the
  route can be guessed or shared freely. Artifacts are **public shareables — don't put
  secrets in one.**
- **No traversal or link escape.** The slug and file path are validated; symlinks,
  hardlinks, dotfiles, and anything resolving outside the `artifacts/` subdir are refused —
  so the public route can't be tricked into serving a sibling source file or a planted link
  to `../.env`. Confinement to `artifacts/` (not the whole workspace) is the protection.
- **Sandboxed pages.** Every artifact is served with `Content-Security-Policy: sandbox`
  (and `X-Content-Type-Options: nosniff`), putting it in a unique opaque origin. Its
  (agent-authored) JavaScript therefore **cannot read the admin origin's `localStorage`** —
  where the admin API key lives — or its cookies. CDN scripts, inline scripts, forms and
  popups still work.
- **Writing asks first.** Publishing an artifact is a `write_file` into the workspace, which
  asks the owner for approval like any other workspace write.
- **Served without authentication** by design — anyone with the link can open it.

## Configuration

```yaml
artifacts:
  enabled: true   # toggles the public /artifacts/ route (files live in the workspace)

workspace:
  enabled: true            # required — provides write_file + the artifacts directory
  directory: "~/pa-workspace"
```

`artifacts.enabled` is managed from the admin **Tools** tab. Setting it `false` 404s every
`/artifacts/<slug>/` link. Serving also requires the **workspace** harness to be enabled —
that's what gives the agent a place to write artifacts in the first place.
